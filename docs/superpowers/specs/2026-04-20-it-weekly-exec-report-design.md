# Дизайн: короткий HTML-отчет для встречи с руководителями

Дата: 2026-04-20
Репозиторий: `/tmp/west-project`
Целевой артефакт: `20-04-2026.html`

## 1. Цель

Собрать короткий, визуально сильный HTML-отчет для встречи с руководителями в том же стиле, что и существующий шаблон `13-10-2025.html`, но адаптированный под недельный формат.

Страница должна:

- быстро давать управленческую картину без перегруза деталями;
- показывать статус двух ключевых проектов: `Склад` и `Сайт`;
- кратко отражать операционную нагрузку IT-отдела;
- содержать прямые ссылки на рабочие источники, в первую очередь на Google Sheet по сайту.

## 2. Формат результата

Реализация будет в виде одной самостоятельной HTML-страницы в корне репозитория:

- имя файла: `20-04-2026.html`;
- без сборки и без бэкенда;
- на базе тех же подходов, что и существующие страницы репозитория: Tailwind CDN, Chart.js CDN, встроенные стили и plain JavaScript;
- пригодна для локального открытия и последующей публикации рядом с другими отчетами.

Это не слайдовая презентация и не многостраничный сайт. Это один длинный executive-отчет с быстрым скроллом и якорной навигацией.

## 3. Источники данных

Контент страницы должен опираться на уже собранные материалы:

- еженедельные DOCX-отчеты IT-отдела за предыдущие недели;
- файл Ганта по проекту склада `Диаграмма_Ганта (1).xlsx`;
- скриншот/дашборд инцидентов с метриками по обращениям и SLA;
- ручные статусы пользователя по проекту сайта;
- Google Sheet по задачам сайта:
  [WEST - Google Sheets](https://docs.google.com/spreadsheets/d/1CZ-ygVjbFeDR7BgL9zayrUSIvyO-gr-FetGvewdDqZE/edit?gid=2137234415#gid=2137234415).

Если в Google Sheet есть ссылки на принты, задачи или домены, они должны быть представлены как компактный блок быстрых ссылок внутри секции сайта, а не как длинная выгрузка таблицы.

## 4. Целевая аудитория и тон

Аудитория:

- руководители;
- участники встречи, которым нужен короткий статус без операционной детализации.

Тон страницы:

- управленческий;
- четкий и спокойный;
- без длинных технических объяснений;
- с акцентом на статус, эффект, риски и решения, которые нужно принять.

## 5. Структура страницы

Страница должна содержать 5-6 сильных секций.

### 5.1 Hero / Executive summary

Верхний экран должен включать:

- бейдж WEST;
- заголовок уровня `Отчет IT-отдела`;
- подзаголовок про итоги недели и ключевые проекты;
- короткую строку с периодом и 1-2 ключевыми фактами;
- крупный summary-блок с абзацем о главном результате недели.

Смысл первого экрана: за 10-15 секунд дать руководителю ответ на вопрос, что именно команда продвинула и где требуются управленческие решения.

### 5.2 KPI / Ключевые показатели

Отдельный визуальный ряд из 4 карточек:

- обращения / инциденты: `532`;
- SLA: `81,02%`;
- проект `Склад`: статус уровня `готовимся к тестированию`;
- проект `Сайт`: статус уровня `визуальная сборка / подготовка макета`.

Карточки должны быть короткими, без тяжелой аналитики, с маленькой подписью, объясняющей смысл числа или статуса.

### 5.3 Фокус недели

Две крупные карточки рядом:

- `Склад`;
- `Сайт`.

В каждой карточке должно быть три блока:

- что сделано;
- что в работе;
- что требуется от руководителей.

Это центральная секция страницы, в которой считывается суть двух основных проектов без перехода в глубину задач.

### 5.4 Инциденты и операционная поддержка

Компактная секция с одним графиком и коротким выводом.

Содержимое:

- простой график по месяцам `Январь / Февраль`;
- акцент на росте нагрузки `215 -> 317`;
- акцент на сохранении SLA `80,00% -> 81,70%`;
- короткая управленческая подпись, что отдел удержал качество при росте объема.

Если визуально уместно, рядом можно показать 2-3 кратких KPI-плашки:

- `431` решено в срок;
- `101` с превышением;
- среднее время выполнения в пределах суток.

### 5.5 Проект «Склад»

Отдельная секция для статуса складского проекта.

Обязательное содержимое:

- короткое описание текущего этапа: завершено проектирование и подготовка, команда подходит к тестированию;
- компактный roadmap по этапам из Ганта;
- статус готовности разработчика к тестированию размещения;
- отдельный callout-блок `Нужно решение`, где перечислены:
  - назначить тестовую команду;
  - дать доступ в копию базы;
  - подтвердить использование базы разработки для первых тестов.

Roadmap должен быть коротким, не превращаться в полноценный календарный план на весь квартал.

### 5.6 Проект «Сайт»

Отдельная секция для сайта с разделением на три трека:

- старый сайт;
- новый сайт;
- сайт учебного центра.

Содержимое:

- по старому сайту: закрыты критичные ошибки;
- по новому сайту: запрошены счет на шаблон и лицензия, переданы референсы;
- по учебному центру: протестированы оплата и передача лида в Bitrix24, стартует визуальная часть.

В секции должна быть заметная CTA-кнопка на Google Sheet задач по сайту.

Дополнительно:

- блок быстрых ссылок на связанные ресурсы из таблицы, если они публичны и полезны на встрече;
- при наличии ссылок на принты они оформляются как компактный список, а не как embedded-галерея.

### 5.7 Решения / Next step

Финальный блок страницы должен содержать 3 коротких решения для руководителей:

- подтвердить тестовую команду по складу;
- не задерживать оплату шаблона и лицензии по новому сайту;
- сохранить быстрый цикл обратной связи по обоим проектам.

Этот блок должен завершать страницу и сводить обсуждение к действиям.

## 6. Визуальное направление

Новая страница должна опираться на язык `13-10-2025.html`, но быть легче и короче.

Сохраняем:

- светлый stone / neutral фон;
- крупную типографику;
- карточную компоновку;
- мягкие тени, градиенты, rounded-блоки;
- sticky-навигацию по якорям.

Адаптируем:

- меньше секций;
- меньше текста в каждом блоке;
- больше воздуха;
- меньше вторичной аналитики;
- больший акцент на executive-summary и project status.

Нельзя превращать страницу в копию шаблона с заменой текста. Нужно сохранить стиль, но переформатировать содержание под недельный отчет.

## 7. Навигация

Верхняя навигация должна быть короткой и соответствовать фактической структуре страницы.

Рекомендуемые пункты:

- `Обзор`;
- `Фокус недели`;
- `Инциденты`;
- `Склад`;
- `Сайт`;
- `Решения`.

Навигация должна работать как плавный переход по якорям и не перегружать верх страницы.

## 8. Данные, которые нужно показать явно

На странице должны быть явно зафиксированы следующие факты:

- `532` обращения;
- `431` решено в срок;
- `101` с превышением;
- общий SLA `81,02%`;
- январь `215`, февраль `317`;
- январский SLA `80,00%`, февральский SLA `81,70%`;
- по складу: разработчик готов начинать первичное тестирование размещения;
- по сайту: критичные ошибки старого сайта исправлены, по новому сайту идет подготовка макета и закупки шаблона/лицензии.

## 9. Ограничения

Нельзя:

- делать длинный отчет на уровне квартального обзора;
- перегружать страницу внутренними задачами сотрудников;
- использовать вымышленные данные;
- строить сложную архитектуру со сборкой или зависимостями вне CDN;
- располагать проект сайта как длинную таблицу со всеми строками Google Sheet.

Нужно:

- показать только те числа и статусы, которые важны на встрече;
- оставить рабочие ссылки в удобном виде;
- сделать страницу пригодной к быстрому показу с ноутбука во время обсуждения.

## 10. Критерии готовности

Страница считается готовой, если:

- в репозитории появляется `20-04-2026.html`;
- файл открывается локально без сборки;
- страница визуально соответствует уровню существующего шаблона;
- ключевые цифры и статусы считываются за 1-2 минуты;
- есть заметная ссылка на Google Sheet по сайту;
- блоки `Склад`, `Сайт` и `Решения` позволяют использовать страницу как опорный сценарий встречи.

## 11. План реализации

После утверждения этой спецификации следующим шагом должен быть отдельный implementation plan, а уже затем сборка HTML-страницы и проверка отображения.
